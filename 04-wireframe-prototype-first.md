# Wireframe-First, Prototype-First

## Why

A page can look polished and still be structurally wrong. Wireframes and prototypes exist to catch structural failure before implementation cost rises.

## Wireframe Stage

Wireframes should answer:

- what is the job of the screen
- what information is primary
- what actions are primary
- what belongs above the fold
- how the mobile layout differs from web

Wireframes should not spend time on:

- final colors
- final typography
- decorative assets

## Prototype Stage

Prototypes should answer:

- how the user moves through the flow
- what changes between states
- where actions live
- whether filters, tabs, and drawers actually make sense
- whether mobile and web both feel coherent

## Required Before Page Contract

Do not write a final page contract until these are stable:

- low-fidelity wireframe
- action flow
- empty/loading/error states
- mobile and desktop hierarchy

## Prototype Fidelity Rule

- low fidelity when structure is still changing
- medium fidelity when the flow is agreed
- higher fidelity only when the interaction model is mostly settled
